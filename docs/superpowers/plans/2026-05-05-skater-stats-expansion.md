# Skater Stats Expansion — Player Profile Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Capture all ~40 missing skater-side metrics from EA's `/members/stats` endpoint, store them in `ea_member_season_stats`, and surface them on the player profile via a 5-tab Club Stats section that matches ChelHead's information density.

**Architecture:** The EA API already returns every metric — our worker calls the endpoint but `transform-members.ts` discards most fields. Plan adds 49 new columns to `ea_member_season_stats` (Drizzle-generated migration), threads the fields through the transform, the `getPlayerEASeasonStats` query, and a new `<ClubStatsTabs>` client component. No new API calls. Goalie expansion (Tabs 6-8) and spatial/hot-zone data are deferred to separate plans.

**Tech Stack:** Drizzle ORM (PostgreSQL), Next.js 15 App Router, TypeScript strict, pnpm workspaces, Turborepo. Existing pipeline: `apps/worker/src/ingest-members.ts` → `apps/worker/src/transform-members.ts` → `ea_member_season_stats`.

**Out of Scope:**
- Goalie Tabs 6, 7, 8 (Goalie Overview / Saves / Situations) — separate plan
- Spatial location data (`skGoalsLocationOnIce*`, `skShotsLocationOnIce*`) — separate plan, requires hot-zone heatmap UI
- Connection-quality metrics that are EA-internal (Quit Disconnects, Forced Completions) — `playerQuitDisc` and `gamesCompletedFC` ARE in the API and will be captured for completeness; the user can decide whether to display them

## Risks and Mitigations

**Risk:** Drizzle migration generation produces a destructive diff (drops/renames an existing column)  
**Mitigation:** Task 1 Step 4 explicitly inspects the migration. If destructive, abort and edit the schema TypeScript file to match what's already in DB.

**Risk:** EA API changes a field name between game titles (e.g. `skhattricks` → `skhattricksMade`)  
**Mitigation:** `parseIntField` returns 0 on missing fields — silent zeros surface as zero values in the UI rather than ingestion failures. The transform is per-field, so a single rename doesn't break the rest.

**Risk:** Numeric fields return as `null` from EA when GP=0, breaking the UI's `${value}%` rendering  
**Mitigation:** Component uses `value !== null ? \`${value}%\` : '—'` for every numeric percentage. Verified in Task 7 Step 1 code.

**Risk:** Other consumers of `getPlayerEASeasonStats` break because of the wider return shape  
**Mitigation:** Only `apps/web/src/app/roster/[id]/page.tsx` reads this query (verified by grep before plan was written). Wider shape is additive — existing fields keep working.

**Risk:** Bundle size grows because the entire ClubStatsTabs component ships to the client  
**Mitigation:** It's a single client component with ~70 stat items inline; tabbed rendering is conditional, so only one grid is rendered at a time. Total client JS impact is small (~5–10KB gzipped). Acceptable for the team-internal audience.

## Out of Scope (Future Plans)

1. **Goalie expansion (Tabs 6, 7, 8)** — separate plan: add ~30 goalie columns (`gldsaves`, `glpokechecks`, `glsoshots`, `glsoperiods`, etc.) and a `<GoalieClubStatsTabs>` component.
2. **Spatial / hot-zone visualization** — separate plan: store the 64 location fields (`skGoalsLocationOnIce1-16`, `skShotsLocationOnIce1-16`, `skGoalsLocationOnNet1-5`, `skShotsLocationOnNet1-5`) as JSONB or a dedicated table, build a rink-overlay heatmap component.
3. **Per-game derived metrics** (gamescore, percentile ranks vs teammates) — derive at query time, no schema change needed.
4. **Backfill of historical NHL titles** — the new columns will be 0 for any pre-existing rows from older titles. If desired, a backfill script can re-call `/members/stats?platform=...` for any title still served by EA.
